## 为什么 UDP 这么重要

先说说背景。我们平时上网用的大多数协议——浏览网页、看视频、刷社交媒体——走的都是 TCP。TCP 有个特点：每丢一个包，它就主动把发送速度减半，这叫"拥塞避免算法"。在网络稳定的时候，TCP 表现很好；但一旦丢包率上去了，比如跨境线路高峰期，TCP 的实际吞吐量会掉得很惨——有时候理论上 100Mbps 的线路，实际传输速度只剩 10-20Mbps。

UDP 不一样。它不管丢包，只管发。所以在高丢包、高延迟的环境下，基于 UDP 的协议反而能维持更稳定的传输。

具体到使用场景上，有几个地方是 TCP 做不到、UDP 才能搞定的：

**游戏联机**：大多数网络游戏的实时同步数据走 UDP，因为游戏需要的是低延迟、不等待，哪怕偶尔丢个包也比等重传要强。用 SStap、Netch 这类工具做游戏加速，UDP 支持是硬要求。

**语音和视频通话**：WhatsApp 电话、VOIP 通话、视频会议，这些都依赖 UDP。TCP 的确认-重传机制在实时语音场景里反而是负担。

**现代代理协议**：Hysteria2、TUIC 等新一代协议都是基于 QUIC，而 QUIC 本质上是跑在 UDP 上的。这类协议在高丢包网络里的表现可以是同等 TCP 协议的两三倍速。

所以当你在搜"搬瓦工 UDP"的时候，背后可能是这几种需求之一：想用搬瓦工搭游戏加速器、想跑 Hysteria2 之类的 QUIC 协议、或者干脆是在排查"代理搭好了但 UDP 就是不通"的问题。

不过有几个地方容易踩坑，提前说清楚：

**坑一：SStap 测试显示 UDP 不通**

有时候用 SStap 测试会提示 UDP 转发失败，但实际上是客户端版本问题或者搭建脚本的问题，不是服务器不支持 UDP。换用酸酸乳（SS-R 的 libev 版本）或者更新一下脚本，一般就解决了。

**坑二：UDP 被运营商 QoS**

这个是国内某些 ISP 的行为，跟搬瓦工本身无关。ISP 在某些时段会限速甚至阻断大流量 UDP 端口。对策是换端口（避开 443，试试 8443 或者 30000+ 的端口），或者改用其他协议。

**坑三：mKCP（KCP over UDP）的流量放大问题**

V2Ray 的 mKCP 协议跑在 UDP 上，可以绕过 TCP 阻断，但有个副作用：它会消耗更多流量。如果你的套餐流量不多，开 mKCP 要注意流量用量。

## 解决 UDP 不通的操作指引

如果你遇到了 UDP 不通的情况，按这个顺序排查：

**第一步，检查服务端防火墙**。很多人忽略了这一步。搬瓦工 VPS 没有外部防火墙，但系统内可能有 iptables 规则在拦截 UDP 流量。用 `iptables -L -n` 查看，如果有拦截 UDP 端口的规则，手动放行：

bash
iptables -I INPUT -p udp --dport 你的端口 -j ACCEPT


**第二步，检查 V2Ray 或协议配置**。V2Ray 的 VMess 对 UDP 的支持"一言难尽"（社区原话），推荐换用 Xray-core，它完整实现了 FullCone NAT，对 UDP 的支持要好得多。

**第三步，检查客户端版本**。SS 客户端版本太老可能不支持 UDP 转发。更新到最新版，或者换用明确支持 UDP 的客户端（如 Shadowrocket、Clash Meta）。

**第四步，换端口重试**。如果特定端口的 UDP 一直不通，很可能是这个端口被 ISP QoS 或者封锁了。换一个高位端口（30000-50000 范围）试试。
